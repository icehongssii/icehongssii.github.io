obsidian 볼트를 깃허브 pprivate 리포에 저장해 쓰고 있는데 사용자가 icehongssii.xyz/어쩌고 요청보내고 내 볼트(=깃허브 repo)에 있는 특정 폴더에 있는 내용을 html로 보여준다. fastapi, aws ecs로 만들었는데..

그런데 생각해보니까 .. 개 븅신짓이었단걸 깨달았다.. 1. 내 블로그 구글에 검색하면 검색 결과도 안나오는데(SEO 설정X), 트래픽이 0에 가깝다고 봐도 되는데 뭣하러 AWS에다가 돈내고 서버를 호스팅 했지? 2. 방문자수가 0에 수렴하는데 왜 redis가 필요하지? 3. 내 블로그에 동적인 요청 처리가 필요하지도 않은데 왜 웹 프레임워크까지 필요한가..?(동적인 처리란, 같은 요청을 해도 요청 시점이나 누가 요청했냐에 따라서 다른 결과를 보여주는건데 쇼핑몰에서 장바구니 담기페이지를 요청하는 경우가 그예이다.) 근데 내 블로그는 언제 누가 요청을 보내든 항상 같은 웹페이지html를 보여주는데? 데이터베이스도 사용하지 않는데 왜 fastapi로 만든 파이썬 어플리케이션이 필요하지..?

그래서 더 단순화가 필요하단걸 깨달았다. 일단 .. 1. redis 필요없음 2. AWS 필요없고 github에다가 곧바로 호스팅 3. fastapi 필요없음

👯‍♂️ 그래서 단순화 버전을 만들었다.

N분마다 개인 obsidian에 있는 내용들이 obsidian-tech-blog github에 푸시된다. 이때 이것들 변환시켜주기 깃허브 액션은 크게 세가지 단계로 이루어져있다

    1. 블로그 폴더 안에 있는 마크다운을 html로 변환
    1. github action artifact 에 인스톨하고 다운로드
    1. github page 리포지토리에 해당 파일들 push

fastapi는 웹 어플리케이션 만드는걸 도와주는 웹 프레임워크인데.. 얘는 asgi 위에서 동작한다. 무슨말이냐면 애초에 파이썬은 웹서버로부터 받은 요청을 곧바로 처리할 수 없기 때문에 중간에 cgi라는 놈을 두는데..이 cgi는 모든 요청에 따라 매번 프로세스를 만들기에 과부하 오기 딱 좋은 놈이다. (이놈의 연장선이 was임) 여튼 이 에바스러운 상황에서 대안으로 나온게 wsgi이다. cgi와 다른 방식으로 요청을 받아서 웹 어플리케이션을 실행한다

프레임워크 없이 웹 애플리케이션을 만든다는 것은 건물을 지을 때 기본적인 철골구조나 규격 없이 처음부터 모든 것을 직접 설계하고 건축하는 것과 비슷합니다.

비유를 조금 더 확장해보자면:

  • 프레임워크가 있는 경우: 건물을 지을 때, 이미 만들어진 철골구조와 규격에 따라 작업합니다. 예를 들어, 벽을 세우거나 창문을 설치할 때 표준화된 치수와 재료를 사용할 수 있습니다. 이렇게 하면 건축 과정이 훨씬 더 빠르고 효율적이며, 결과물의 품질도 일정하게 유지됩니다. 이처럼, 웹 프레임워크는 개발자가 복잡한 작업을 쉽게 할 수 있도록 기본 구조와 규칙을 제공하여, 더 빠르고 안정적으로 웹 애플리케이션을 만들 수 있게 도와줍니다.

  • 프레임워크가 없는 경우: 반면, 프레임워크 없이 건물을 짓는다면, 철골부터 전기 배선까지 모든 것을 직접 설계하고 만들어야 합니다. 이 과정에서 발생할 수 있는 오류도 많고, 시간이 많이 소요될 뿐만 아니라 결과물의 품질도 보장하기 어렵습니다. 마찬가지로, 웹 프레임워크 없이 애플리케이션을 만든다면, HTTP 요청을 처리하거나 데이터베이스와 상호작용하는 등의 기본적인 기능을 모두 처음부터 직접 구현해야 합니다. 이는 많은 시간과 노력이 필요하며, 코드의 복잡성도 증가하게 됩니다.

결론:

프레임워크는 건축에서의 철골구조처럼, 웹 애플리케이션 개발에서의 기본 구조와 규격을 제공합니다. 이를 통해 개발자는 반복적인 작업을 줄이고, 더 빠르고 안정적으로 애플리케이션을 개발할 수 있습니다. 만약 이러한 프레임워크가 없다면, 개발자는 모든 것을 처음부터 구현해야 하므로, 개발 속도가 느려지고 오류 가능성이 높아지며 유지보수가 어려워질 수 있습니다.
